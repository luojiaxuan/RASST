# GigaSpeech-only HN1024 varctx data ablation

## Hypothesis

Removing `wiki_synth` training rows from the current `lh1b88kw` main-result
recipe isolates whether the HN1024 retriever gains most of its held-out recall
from real GigaSpeech acoustic evidence or from synthetic wiki term coverage.

## Background / Motivation

The current main result is W&B run `lh1b88kw`:
`variantE_hn1024_gsv2full_gsdedup_varctx576_v3_bs8k_gc128_tcmoff_ep6_8gpu_aries`.
Its training set mixes GigaSpeech and `wiki_synth` rows.  The dataset stats for
the mixed varctx576 JSONL report about 4.66M GigaSpeech rows and 2.89M
`wiki_synth` rows after repair.

## What changed vs baseline

- Training data changes from the mixed varctx576 JSONL to a filtered
  GigaSpeech-only JSONL generated by
  `documents/code/data_pre/training_terms_for_retriever/filter_gigaspeech_only_jsonl.py`.
- The train/eval context durations remain 2.88s, 3.84s, 4.80s, and 5.76s with
  `fixed_audio_seconds=5.76`.
- HN depth, encoders, LoRA ranks, MaxSim MFA, and TCM-off settings stay aligned
  with the `lh1b88kw` recipe.
- Runtime uses Aries GPUs 0,1,2,3 once they are all idle.  The 4-GPU run keeps
  global batch size 8192 via GradCache chunk 128 to stay close to the main
  result under the smaller GPU budget.

## Expected metrics

Dev `recall@10_gs10000` may drop if synthetic wiki rows were carrying rare-term
coverage.  ACL and medicine readouts are kept as monitoring/readout only; this
run should not use ACL to choose checkpoints or hyperparameters.
